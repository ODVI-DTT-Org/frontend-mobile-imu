# Offline Client Writes Design

## Goal

Queue client create, update, and delete operations locally when offline and sync them to the server when connectivity is restored — using the same Hive-queue pattern as pending visits and releases.

## Architecture

```
AddClientPage / EditClientPage / delete trigger
       ↓
ClientMutationService          ← single entry point for all client writes
  ├─ Online  → ClientApiService (existing) → Hive cache update
  └─ Offline → PendingClientService.enqueue() + optimistic Hive cache update
                       ↓
             Hive box: 'pending_clients'
                       ↓
     BackgroundSyncService._syncPendingClients()
             (fires on connectivity restore)
                       ↓
             collapse queue per clientId
                       ↓
             ClientApiService (create / update / delete)
                       ↓
             clean up pending entry + update Hive cache
```

## Tech Stack

Hive (offline queue), Riverpod (providers), ConnectivityService (online/offline check), existing ClientApiService, existing HiveService, existing BackgroundSyncService pattern.

## Data Model

**File:** `lib/services/client/models/pending_client_operation.dart`

```dart
enum ClientOperationType { create, update, delete }

class PendingClientOperation {
  final String id;           // UUID for this queue entry
  final ClientOperationType operation;
  final String clientId;     // temp UUID for creates, real server ID for update/delete
  final Map<String, dynamic>? clientData;  // null for delete ops
  final DateTime createdAt;

  PendingClientOperation({...});

  Map<String, dynamic> toJson();
  factory PendingClientOperation.fromJson(Map<String, dynamic> json);
}
```

`clientId` for offline creates is a temp `uuid.v4()` generated by `ClientMutationService`. It serves as the Hive key so the client appears in the UI immediately. On sync:
- Admin path: server returns real client — swap temp Hive entry for real one.
- Caravan path: server returns `requires_approval: true` — remove temp Hive entry; PowerSync brings in the real record after admin approval.

## Collapse Rules

Applied per `clientId` before syncing, processed in `createdAt` order:

| Queue contents for a clientId | Collapsed result |
|---|---|
| create → update(s) | single create with final data |
| create → delete | nothing — cancel both, never touch server |
| update(s) | single update with final data |
| update → delete | single delete |
| delete alone | delete |

## Services

### PendingClientService

**File:** `lib/services/client/pending_client_service.dart`

Manages the `pending_clients` Hive box.

```dart
class PendingClientService {
  static const String _boxName = 'pending_clients';

  Future<Box<Map>> _getBox();
  Future<void> enqueue(PendingClientOperation op);
  Future<List<PendingClientOperation>> getAll();      // sorted by createdAt
  Future<void> remove(String id);
  Future<int> getPendingCount();
  List<PendingClientOperation> collapse(List<PendingClientOperation> ops);
}
```

`collapse()` is a pure function — takes the sorted list, returns the minimal list of operations to send to the server, applying the rules above.

### ClientMutationService

**File:** `lib/services/client/client_mutation_service.dart`

Single entry point for all client writes. Checks connectivity and routes accordingly.

```dart
enum ClientMutationResult { success, requiresApproval, queued }

class ClientMutationService {
  final ConnectivityService _connectivity;
  final ClientApiService _api;
  final PendingClientService _pending;
  final HiveService _hive;
  final Uuid _uuid;

  Future<ClientMutationResult> createClient(Client client);
  // Online: call _api.createClient(); update Hive on success.
  // Offline: assign temp UUID to client, save to Hive, enqueue create op.
  // Returns success | requiresApproval | queued.

  Future<ClientMutationResult> updateClient(Client client);
  // Online: call _api.updateClient(); update Hive on success.
  // Offline: save updated client to Hive (optimistic), enqueue update op.
  // Returns success | requiresApproval | queued.

  Future<void> deleteClient(String clientId);
  // Online: call _api.deleteClient(); remove from Hive.
  // Offline: remove from Hive immediately, enqueue delete op.
}
```

## Page Changes

**`add_client_page.dart`** and **`edit_client_page.dart`**: replace inline `isOnline` branching with `ClientMutationService`. Handle three result states:

```dart
final result = await mutationService.createClient(newClient); // or updateClient

switch (result) {
  case ClientMutationResult.success:       showSuccess('Client saved');
  case ClientMutationResult.requiresApproval: showInfo('Submitted for approval');
  case ClientMutationResult.queued:        showInfo('Offline — will sync when connected');
}
```

**Delete trigger** (client detail or edit page): call `mutationService.deleteClient(clientId)` — no return value, show feedback based on connectivity state.

## Background Sync

**`background_sync_service.dart`** — new `_syncPendingClients()` method:

```
1. getAll() from PendingClientService
2. If empty, return early
3. collapse(ops)
4. For each collapsed op:
   create  → POST via ClientApiService
             if real ID returned (admin): delete temp Hive entry, save with real ID
             if requires_approval (Caravan): delete temp Hive entry (PowerSync will sync later)
             remove from pending queue
   update  → PUT via ClientApiService
             update Hive on success
             remove from pending queue
   delete  → DELETE via ClientApiService
             remove from Hive
             remove from pending queue
5. On per-op error: log + continue (leave in queue for next sync attempt)
```

`performSync()` gains: `await _syncPendingClients()`

`_updatePendingCount()` gains: `+ await PendingClientService().getPendingCount()`

## Provider Registration

**`app_providers.dart`** — add:

```dart
final pendingClientServiceProvider = Provider<PendingClientService>((ref) {
  return PendingClientService();
});

final clientMutationServiceProvider = Provider<ClientMutationService>((ref) {
  return ClientMutationService(
    ref.read(connectivityServiceProvider),
    ref.read(clientApiServiceProvider),
    ref.read(pendingClientServiceProvider),
    ref.read(hiveServiceProvider),
  );
});
```

**Hive box registration** — open `pending_clients` box in app initialization alongside `pending_visits` and `pending_releases`.

## Error Handling

- Per-operation sync errors are logged and the entry stays in the queue for the next sync attempt.
- If a delete fails (e.g., client not found on server — 404), treat as success and remove from queue.
- If a create fails with a non-retriable error (e.g., 422 validation), log the error, remove from queue, and remove the temp Hive entry to avoid stale data.
